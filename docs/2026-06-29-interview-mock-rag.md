## 第一阶段：开场与项目概述（约 8 分钟）

### A：你好，我是今天的技术面试官。先简单做个自我介绍吧，重点说一下你最近做的项目，还有你在项目里主要负责的部分。

### B：

嗯好的，面试官您好。我叫 XXX，最近这几个月主要在做一个叫 TalentSense HR 的项目，是一个基于 AI 的智能招聘全栈系统。简单说就是把传统招聘系统那种关键词匹配的方式，升级成用大模型加向量检索做语义推荐，让 HR 可以用自然语言去描述需求，比如"帮我找个 5 年经验的 Java 后端，本科以上，30K 以内"，系统能理解这个语义然后推荐合适的候选人。

这个项目是我独立完成的，从前到后全栈都做。后端用的是 FastAPI，Python 3.12，三层架构；前端是 Vue3 加 TypeScript；数据存储用了 MongoDB、Redis、Milvus、MinIO 这四个；AI 部分用了通义千问 qwen-plus 做大模型，BGE-M3 做向量化，BGE-Reranker 做精排，然后用 LangGraph 编排 Agent 工作流。整体代码量大概两万行，测试有 464 个全通过，最后用 Docker Compose 一键部署，7 个容器编排。

我在里面主要负责的就是整个系统的架构设计和技术选型，特别是 RAG 检索这块的核心链路设计，包括 混合检索、Reranker 精排、还有后来做的 RAG 三模块增强，就是查询分解、多路召回、上下文压缩这三个模块。

### A：嗯，听起来挺完整的。你刚才提到 RAG 三模块增强，这个我们待会详细聊。先问一下，你为什么选择做这个项目？就是市面上招聘系统已经很多了，你做这个的出发点是什么？

### B：

对，这个我有想过。其实出发点主要是两个痛点吧。

第一个是我在调研的时候发现，传统招聘系统，包括一些大厂的，都还是基于关键词的精确匹配。HR 输入"前端工程师"，系统只会去简历里找有没有"前端"这两个字。但实际简历里很多人写的是"Vue"、"React"、"JavaScript"，这种语义关联传统匹配是做不了的。这就导致 HR 要么漏掉很多合适的候选人，要么得手动输一堆关键词去搜，效率很低。

第二个是评分黑箱的问题。传统系统给候选人打个 85 分、60 分，但是为什么是这个分，HR 完全不知道。这个对于招聘决策其实是很不友好的，因为 HR 需要的是"为什么推荐这个人"，而不只是一个冷冰冰的分数。

所以我就想用大模型加向量检索来解决这个问题。向量检索能做语义匹配，"前端"能匹配到"Vue/React"；然后大模型可以做结构化评分，把技能、经验、学历、薪资拆成四个维度，每个维度都有分，还带推荐理由，这样 HR 就能看懂每个分数了。

### A：OK，痛点讲得挺清楚。那在你做这个项目之前，你有调研过市面上的 RAG 方案吗？就是你是怎么从 0 到 1 把技术方案定下来的？

### B：

有的，我调研了不少。主要看了几个方向吧。

一个是 LlamaIndex 和 LangChain 这两个框架，它们都有现成的 RAG 实现。但是我后来没有直接用，因为我觉得它们的 RAG 链路比较通用，但是对于招聘这种特定场景，我需要更精细的控制，比如父子块切分、混合检索的权重、Reranker 的接入点，这些用通用框架反而不好定制。

然后向量化模型我对比了几个，OpenAI 的 text-embedding-3，还有 BGE 系列。最后选 BGE-M3 主要是因为它是开源的，中文效果好，最关键的是它能同时输出稠密和稀疏向量，这对混合检索很重要，一次编码就能满足两个需求。

Agent 框架我看了 LangGraph 和 AutoGen，最后选 LangGraph 是因为它的 State 管理和条件路由比较清晰，而且跟 LangChain 生态兼容，虽然没有直接用 LangChain 的 RAG 链，但是 LLM 客户端这些还是复用了。

整体方案我是先定了核心检索链路，就是向量化、Hybrid 检索、Reranker 精排、LLM 评分这条主线，然后再往外扩展，加 Agent、加邮件、加看板这些业务功能。这样保证核心是扎实的。

## 第二阶段：RAG 核心检索链路深挖（约 15 分钟）

### A：好，那我们就进入正题，聊聊 RAG 这块。你先详细讲一下你的检索链路，从用户输入一个 query 到最后返回候选人，中间完整经过哪些步骤？

### B：

好的，我完整的讲一下。整个检索链路大概是这样的：

用户输入一个 query，比如"5 年 Java 后端本科以上 30K 以内"。第一步是策略选择，我会用一个 LLM 来判断这个 query 应该用哪种改写策略。我有四种策略，直接检索，生成假设答案再检索，拆成子查询，以及简化历史问题。这一步是 LLM 一次调用返回策略名。

选完策略之后是查询改写，根据策略生成一个或多个改写后的 query。比如 拆成子查询策略可能会把"5 年 Java 后端"拆成"5 年后端开发"和"Java 工程师"这样的子查询。

然后是向量化，用 BGE-M3 对改写后的 query 编码，每个 query 同时输出稠密向量和稀疏向量。

接下来是混合检索，我把稠密向量和稀疏向量一起送到 Milvus，用 加权排序器(WeightedRanker) 做加权融合，稠密权重 1.0，稀疏权重 0.7，召回 Top 20 的 chunks。

召回之后是 chunk 去重，因为同一个 chunk 可能被多个改写 query 命中，要去重。

然后是 Reranker 精排，用 BGE-Reranker-v2-m3 对去重后的 chunks 重新打分排序，取 Top 10。Reranker 使用 交叉编码器精排阶段 (cross-encoder)，精度比双塔向量（Bi-Encoder）检索高，但是慢，所以先用向量检索粗筛再用 Reranker 精排。

精排之后是简历级去重，这个挺关键的。因为我是按 chunk 入库的，同一份简历可能被切成多个 chunks，如果不去重，同一份简历的多个 chunks 会挤占结果，导致返回的候选人数量少。所以我按 简历ID(resume_id) 去重，每个 简历ID(resume_id) 只保留 简历分数(rerank_score) 最高的那个 chunk。

去重之后是硬过滤，用 Python 内存层过滤掉不满足硬条件的，比如学历、年限、薪资、技能这些。

最后是 LLM 评分，对过滤后的候选人用 LLM 打分，输出 技能(skill)、工作经验(experience)、学历(education)、(salary) 四个维度的分，加一个 overall 综合分和 reason 推荐理由。评分用 asyncio.gather 并发执行，总耗时约等于单次耗时。

评分完之后按 overall 降序返回候选人列表，同时把结果缓存到 Redis，TTL 5 分钟，相同查询直接命中缓存。

### A：你刚才提到 Hybrid 检索，稠密权重 1.0 稀疏权重 0.7，这个权重你是怎么定的？有没有做过实验？

### B：

这个权重说实话，一开始是凭经验定的。稠密向量主要负责语义召回，"前端"能匹配到"Vue/React"；稀疏向量负责关键词精确匹配，"Java"必须命中。我当时想的是语义召回为主，关键词精确为辅，所以稠密给 1.0，稀疏给 0.7。

实验我有做过一些，但不是特别系统的 A/B 测试。我试过 1.0:0.5、1.0:0.7、1.0:1.0 这几个组合，用一批测试 query 跑，人工评估召回的相关性。最后发现 1.0:0.7 这个组合在我这个场景下效果最好，1.0:1.0 的话稀疏权重太高，会把一些关键词命中但语义不相关的结果排到前面；1.0:0.5 又会让稀疏的作用太弱，一些精确技能匹配的要求满足不了。

不过这个权重我放在 .env 配置里了，是可以调的。如果后续有更多真实数据，可以做更系统的 A/B 测试来优化。

### A：嗯，权重可配置这点做得不错。我再问一个细节，你的稠密向量和稀疏向量是怎么从 BGE-M3 一起拿到的？BGE-M3 的 encode 接口是怎么用的？

### B：

BGE-M3 这个模型它原生支持多向量化输出。调用 encode 的时候传几个参数，return_dense=True、return_sparse=True，它会返回一个字典，里面有 dense 字段和 sparse 字段。

dense 是一个 1024 维的浮点数组，就是标准的语义向量。sparse 是一个稀疏表示，格式是 {token_id: weight} 这样的字典，它会把文本里的重要词抽取出来赋一个权重。

我封装在 core/embedding.py 里，提供了一个 encode 方法，输入文本列表，返回 (dense_vectors, sparse_vectors) 两个列表。调用方不用关心 BGE-M3 的细节。

有一个坑我想提一下，就是 BGE-M3 的 encode 返回的是 numpy 类型，比如 numpy.float32、numpy.ndarray。这个直接给 Milvus 用会报 protobuf 序列化错误，因为 Milvus 底层是 protobuf，不认 numpy 类型。所以我在 vector_store.py 里做了一层统一转换，把 numpy.float32 转成 Python 原生 float，numpy.ndarray 转成 list。这个坑当时排查了挺久。

### A：对，这个 numpy 类型的坑确实常见。那你稀疏向量存到 Milvus 是怎么存的？Milvus 的 sparse 字段你是怎么定义和建索引的？

### B：

Milvus 2.4 之后原生支持 SPARSE_FLOAT_VECTOR 这个字段类型。我在 Collection schema 里定义了一个 sparse_vector 字段，类型是 SPARSE_FLOAT_VECTOR。

索引这块，稀疏向量用的是 SPARSE_INVERTED_INDEX，metric_type 用 IP，就是内积。稠密向量用的是 IVF_FLAT，metric_type 也是 IP，nlist 设的 1024。

Hybrid 检索的时候，我用 AnnSearchRequest 分别构建 dense 和 sparse 的检索请求，然后用 WeightedRanker 把两个请求的结果加权融合，最后 limit 取 Top 20。这个是 Milvus 原生的 hybrid_search 接口支持的，不用自己在 Python 层合并。

### A：OK，Hybrid 这块讲得挺清楚。那你刚才提到父子块切分，这个能详细讲一下吗？为什么要做父子块，而不是直接切固定大小的块？

### B：

好的。父子块切分主要是为了解决一个矛盾：检索的时候希望块小一点，这样向量精度高，能精确匹配到某一段内容；但是给 LLM 评分的时候希望上下文大一点，不然只给一个 300 字的片段，LLM 没法判断这个候选人的完整情况。

所以我的做法是，把简历文本切成两层。子块 300 字符，用于入库 Milvus 做向量检索，保证检索精度。父块 1200 字符，作为 parent_content 字段一起存到 Milvus，但是不入向量索引。

检索的时候，命中的是子块，但是我通过 parent_id 字段把对应的父块内容也取出来，作为 LLM 评分的上下文。这样既保证了检索精度，又给 LLM 提供了足够的上下文信息。这个思路业界叫 Small-to-Big，就是检索小块，使用大块。

实现上我在 utils/chunker.py 里写了 split_parent_child 方法，输入是文本，输出是 (child_chunks, parent_chunks)。子块带 parent_id 指向父块，父块带完整内容。入库的时候子块的字段都填上，parent_content 也冗余存一份，避免检索完还要二次查父块。

### A：这个设计挺合理的。那我想问一下，你切块的时候有没有考虑过重叠？就是 overlap，相邻块之间重叠一部分，避免把完整信息切断。

### B：

有的，我有考虑。但是当时权衡之后没有做 overlap。原因是简历这种文本它的结构其实比较清晰，比如基本信息、教育经历、工作经历、项目经历、技能，这些段落之间边界比较明确。我用的是按段落加长度混合的切分方式，尽量在段落边界切，不会把一段从中间切断。

不过这个确实是个 trade-off。如果做 overlap，比如 50 字符的重叠，能避免一些边界信息丢失，但是会增加存储量和检索时的冗余命中。我当时觉得简历文本结构清晰，不做 overlap 也能接受。如果是那种长文档、段落边界不清晰的场景，比如论文、技术文档，overlap 是必要的。

### A：嗯，有思考就好。那我们继续往下走，Reranker 这块。你用 BGE-Reranker-v2-m3，为什么需要 Reranker？向量检索不是已经能召回了吗？

### B：

这个是个很好的问题。向量检索，不管是稠密还是稀疏，本质都是双塔模型，就是 query 和 doc 分别独立编码，然后用向量相似度计算相关性。这种方式的好处是快，可以离线把所有 doc 编码好建索引，线上只编码 query 就能检索。但是缺点是精度有限，因为 query 和 doc 是独立编码的，它们之间的交互只发生在最后的点积这一步，没有深度的特征交互。

Reranker 是 cross-encoder，它把 query 和 doc 拼接在一起送进模型，模型能在编码的时候就看到 query 和 doc 的关系，能做深度的特征交互，所以精度高很多。但是缺点是慢，因为每个 doc 都要和 query 一起过一遍模型，没法离线预处理。

所以业界标准做法就是两阶段：第一阶段用向量检索粗筛，召回 Top 20 或者 Top 50；第二阶段用 Reranker 精排，从粗筛结果里选 Top 10。这样兼顾速度和精度。

我项目里就是先 Hybrid 检索召回 Top 20，然后用 BGE-Reranker-v2-m3 精排取 Top 10。Reranker 的输入是 query 和每个 chunk 的 parent_content，输出是一个相关性分数，我按分数降序取前 10。

## 第三阶段：RAG 三模块增强深挖（约 20 分钟，重点）

### A：OK，基础检索链路讲清楚了。接下来聊你刚才提到的 RAG 三模块增强，这是你后来迭代加的。你先说说为什么要做这个增强？原来的链路有什么问题？

### B：

好的，这个其实是从一个真实 bug 引出的。当时我在测试的时候发现，搜"NLP 相关的候选人"，只返回了 3 到 5 个，但是我库里其实有 13 份 NLP 相关的简历。我就去排查，发现有好几个问题。

第一个问题是 chunk 级截断。我的 Reranker 精排之后取 Top 10，这个 Top 10 是 chunk 级别的，不是简历级别的。同一份简历如果切成了 5 个 chunks，其中 3 个都被召回了，那这 3 个 chunks 会占据 Top 10 里的 3 个位置，导致最终返回的独立简历数量很少。这个我后来通过简历级去重解决了，就是按 resume_id 去重，每个 resume_id 只保留最高分的 chunk。

第二个问题是 Milvus 的标量字段没有存真实值。我建 Collection 的时候定义了 salary_min、education_level、work_years 这些标量字段，用于过滤，但是入库的时候这些字段都写死了 0 或者 1，没有存真实的值。这就导致过滤功能完全失效，"本科以上"这种过滤条件根本不起作用。这个我后来通过修改 vector_store.py，入库的时候把真实元数据存进去，还写了一个 reindex 脚本重建了索引。

第三个问题是 NLP 这个词和 BERT、Transformer、FastText 这些具体技术词，在向量空间里相似度不高。HR 搜"NLP 相关的"，其实是想找所有做自然语言处理的人，包括简历里写 BERT、写 Transformer 的。但是稠密向量把 NLP 和 BERT 编码出来的向量相似度不够高，BM25 稀疏向量又只看词面，NLP 和 BERT 词面完全不同，所以漏召回很严重。

第四个问题是复杂多条件查询。比如"3 年经验后端工程师会 Python 和 Docker 30K 以内"这种 query，作为一整个查询去编码向量，所有条件被压缩到一个向量里，相似度被稀释，召回率很低。

这几个问题加起来，就导致检索效果不理想。所以我做了 RAG 三模块增强，分别解决这些问题。

### A：嗯，你这个从 bug 驱动优化的思路很好。那你分别讲一下这三个模块，先讲查询分解。

### B：

好的。查询分解这个模块叫 query_decompose_node，它替代了原来的 filter_extract_node。

原来的 filter_extract_node 是用 LLM 从 query 里提取硬过滤条件，比如学历、年限、薪资、技能，然后把这些条件存到 state.filters 里，后面硬过滤的时候用。但是它只提取条件，不改写 query。

我做的查询分解是它的超集，LLM 输出三部分：main_query、sub_queries、structured_filters。main_query 是主查询，sub_queries 是子查询列表，最多 3 个，structured_filters 是结构化过滤条件。

举个例子，输入"3 年经验后端工程师会 Python 和 Docker 30K 以内"，LLM 会输出：
- main_query: "后端工程师 Python Docker 3 年经验 30K 以内"
- sub_queries: ["后端工程师 Python Docker", "3 年经验后端开发", "Python Docker 微服务"]
- structured_filters: {required_skills: ["python", "docker"], work_years_min: 3, salary_max: 30, job_type: "后端"}

sub_queries 用于多路召回，每个子查询独立检索，解决多条件被向量稀释的问题。structured_filters 合并到 filters 走硬过滤保底。

这个节点有一个兜底机制，LLM 调用失败的时候，我会用正则提取硬过滤条件。正则能识别"X年以上"、"XK以内"、"会X的"这些模式。虽然不如 LLM 精确，但是能保证基本条件不丢失。

### A：你这里说 sub_queries 最多 3 个，为什么是 3 个？多了会怎样？

### B：

这个是权衡过的。子查询的作用是让每个子条件独立检索，提升召回。但是子查询越多，检索次数越多，延迟越高，而且结果合并去重的开销也大。

我测下来 3 个子查询是个比较好的平衡点。一个复杂 query 拆成主查询加 3 个子查询，4 路检索并发执行，延迟增加不多，但是召回率提升明显。

如果拆 5 个甚至更多，一方面 LLM 拆出来的子查询质量会下降，可能会拆出一些没什么意义的查询；另一方面检索结果太多，Reranker 精排的输入变大，延迟会明显增加。所以我硬限制最多 3 个，在代码里 [:3] 截断。

### A：OK，那讲第二个模块，多路召回。

### B：

好的。多路召回是搜索的核心增强，分三路。

Route A 是主查询的 Hybrid 检索，就是用 main_query 做 Dense 加 Sparse 的混合检索，这是原有的逻辑，负责语义召回。

Route B 是子查询的独立检索，对每个 sub_query 单独做 Hybrid 检索。这一路解决的是多条件被向量稀释的问题。比如"3 年经验 Python Docker"这个 query，整体编码向量的时候，"3 年经验"、"Python"、"Docker"三个条件被压缩到一个向量里，每个条件的权重都被稀释。拆成子查询之后，每个子查询只关注一部分条件，向量更聚焦，召回更精准。

Route C 是 MongoDB 结构化精确匹配，这一路是保底用的。我用 structured_filters 去 MongoDB 直接查，比如 required_skills 用正则匹配 skills 字段，work_years_min 用 $gte，salary_max 用 $lte。查到匹配的 resume_id 之后，再去 Milvus 拉这些简历的 chunks，赋予 rerank_score=0 兜底，让 Reranker 重新打分。

Route C 这个设计主要是为了解决我刚才说的 NLP 和 BERT 的问题。HR 搜 NLP，库里有人简历写的是 BERT、Transformer，向量检索召回不到，但是 MongoDB 里 skills 字段如果做了同义词扩展，就能查到。这个同义词扩展我在 utils/skill_synonyms.py 里维护了一个映射表，nlp 会扩展成 [nlp, bert, fasttext, transformer, jieba] 这些，查 MongoDB 的时候用 $or 把所有同义词都查出来。

三路结果合并的时候按 chunk_id 去重，然后统一送 Reranker 精排。

### A：等一下，你 Route C 说去 MongoDB 查 skills 字段，但是你的 structured_filters 里的 required_skills 是从 query 里提取的，HR 搜"NLP"的时候，required_skills 就是 ["nlp"]，那你去 MongoDB 查 skills 包含 "nlp" 的，这不还是查不到写 "BERT" 的人吗？同义词扩展是在哪一步做的？

### B：

啊，您说得对，我刚才讲得不够清楚。同义词扩展是在 Route C 查 MongoDB 之前做的。

具体流程是这样的：query_decompose_node 输出 structured_filters，里面有 required_skills，比如 ["nlp"]。然后到 Route C 的时候，我先调一个 expand_skills 函数，把 ["nlp"] 扩展成 ["nlp", "bert", "fasttext", "transformer", "jieba"]。然后用扩展后的列表去 MongoDB 查，skills 字段用 $or 加 $regex 匹配任意一个词。

这样 HR 搜 NLP，库里写 BERT 的候选人就能通过 Route C 召回。

不过这里有个设计权衡，就是同义词扩展只对领域词做，不对具体技术词做。比如 nlp 是领域词，扩展成 BERT、Transformer 这些；但是 html 是具体技术，就不扩展，搜 html 只返回真的会 html 的人。这个区分是为了避免过度召回，搜 html 的时候返回一堆会 vue 的人就没意义了。

同义词映射表是我手工维护的，在 utils/skill_synonyms.py 里，目前有 nlp、frontend、backend 这几个领域词的映射。后续可以用 LLM 自动扩展，但是目前手维护的准确率更高。

### A：嗯，这个设计有思考。但是我想问一下，你这个同义词表是手工维护的，那如果有新的技术栈出现，比如突然流行起一个新的框架，你的表不更新就召回不到，这个问题你怎么看？

### B：

您说得对，这确实是个局限。手工维护的同义词表覆盖面有限，而且跟不上技术更新。

我的想法是分两步走。短期就是我定期更新这个表，关注主流技术栈的变化。长期的话，可以用 LLM 做动态扩展。就是在 Route C 之前，把 required_skills 送给 LLM，让 LLM 根据当前技术生态生成同义词列表。比如输入"nlp"，LLM 返回 ["bert", "transformer", "fasttext", "jieba", "spacy", "huggingface"] 这样。

但是 LLM 动态扩展有成本，每次查询都要调一次 LLM，延迟会增加。所以可以加一层缓存，把常见技能的同义词列表缓存到 Redis，TTL 设长一点，比如一天。新技能第一次查询的时候走 LLM，后续命中缓存。

这个目前还没实现，是个 TODO，但是架构上预留了扩展点，就是 expand_skills 这个函数，现在读静态表，以后改成读缓存或者调 LLM 都可以。

### A：好，那继续讲第三个模块，上下文压缩。

### B：

好的。上下文压缩这个模块叫 context_compress_node，它解决的是 LLM 评分信息不完整的问题。

原来的流程是，Reranker 精排之后，对每个简历只取 rerank_score 最高的那个 chunk，然后把它的 parent_content 作为 LLM 评分的上下文。但是问题是，一份简历可能多个 chunks 都被召回，每个 chunk 包含的信息不一样。比如一个 chunk 是技能列表，一个 chunk 是项目经验，一个 chunk 是工作经历。如果只取最高分的 chunk，可能只取到了技能列表，LLM 评分的时候看不到项目经验，评分就不准。

所以我做了上下文压缩。在 Reranker 精排之后、硬过滤之前，对同一 resume_id 的多个 chunks，用 LLM 合并压缩成一个单一的精炼 context，控制在 500 字以内。

具体来说，我按 resume_id 分组，每个分组里把 chunks 按 rerank_score 降序排列，把它们的 parent_content 用换行符拼接起来，送给 LLM，prompt 要求 LLM 提取跟 query 相关的关键信息，包括技能匹配点、经验匹配点、项目匹配点，输出一个 500 字以内的精炼摘要。

这个压缩后的 context 会替换掉原来的 parent_content，传给后面的 LLM 评分环节，让评分基于更完整的信息。

实现上有个细节，就是只对命中 2 个及以上 chunks 的简历做压缩，单 chunk 的不压缩，避免浪费 LLM 调用。然后压缩是并发执行的，用 asyncio.gather，多个简历并行压缩，总耗时约等于单次 LLM 调用。

兜底机制是 LLM 压缩失败的时候，回退到取最高分 chunk 的 parent_content，就是原来的逻辑，保证不会因为压缩失败导致整个检索挂掉。

### A：你这里说压缩后控制在 500 字以内，这个 500 字是怎么定的？有没有考虑过 token 数而不是字数？

### B：

500 字是凭经验定的，主要考虑两个因素。一个是 LLM 评分的输入不能太长，否则延迟高且贵。我评分的时候是多个候选人一起送给 LLM，如果每个候选人的 context 都 1000 字，10 个候选人就是 10000 字，加上 prompt 和 query，输入 token 就很多了。500 字的话 10 个候选人 5000 字，比较可控。

另一个是 500 字足够覆盖关键信息。简历里跟一个 query 相关的内容，技能匹配点、经验匹配点、项目匹配点，压缩到 500 字是够的，不需要把整份简历都给 LLM。

您说的 token 数确实更准确，因为中文字符和 token 不是一一对应。我用字数主要是简单，而且 LLM 的 context window 足够大，500 字对应的 token 数不会超限。如果要做更精细的控制，可以用 tiktoken 或者 LLM 提供的 tokenizer 来数 token，但是当前场景字数够用了。

### A：OK。那我想问一个整体的问题，你这三个模块加起来，延迟增加了多少？你觉得值得吗？

### B：

这个我测过。查询分解多一次 LLM 调用，大概 2 秒。上下文压缩多一次 LLM 调用，因为是并发的，大概也是 2 秒。多路召回本身不增加延迟，因为三路是并发的。所以总共增加大概 4 秒左右。

原来一次检索大概 5 秒，增强后大概 9 秒。延迟增加了 80%，确实不算少。

但是我觉得值得，因为质量提升很明显。NLP 相关查询从 3 到 5 个候选人提升到 13 个全召回，多条件查询的召回率提升 40% 以上。而且我这个项目的定位是 HR 招聘场景，HR 用这个系统是想找到合适的候选人，质量比速度重要。HR 愿意为找到对的人多等几秒。

不过我也做了优化，就是缓存。相同 query 的检索结果缓存到 Redis，TTL 5 分钟，第二次查询直接命中缓存，延迟降到 200 毫秒以内。所以对于热门查询，延迟问题不大。

如果后续要进一步优化延迟，可以考虑几个方向。一个是查询分解和上下文压缩用更快的模型，比如 qwen-turbo 而不是 qwen-plus，牺牲一点质量换速度。另一个是压缩这一步可以用一个小模型本地跑，不调 API，这样延迟更可控。这些都是后续优化方向。

### A：你提到缓存，我想多问一句。你的缓存 key 是怎么设计的？如果简历库更新了，缓存怎么失效？

### B：

缓存 key 我用的是 query 加上 strategy 加上 filters 的字符串表示，再加 top_k，然后做一次 hash。比如 `search:{hash((query, strategy, str(filters), top_k))}`。这样相同查询条件命中同一个缓存。

失效策略我用的是 TTL 自然过期，设的 5 分钟。没有做主动失效，因为主动失效要精确清除某个 query 的缓存，但是 key 是 hash 的，没办法反查。所以就用短 TTL，5 分钟内简历库更新了，最多 5 分钟后缓存自然过期，重新检索。

这个策略简单但是有缺点，就是 5 分钟内如果有新简历上传，用户搜不到，要等缓存过期。对于我这个场景，简历上传频率不高，这个问题不严重。如果是高频更新场景，可以用更复杂的失效策略，比如维护一个 query 到 cache key 的反向索引，简历更新的时候批量清除相关 query 的缓存。

### A：嗯，有取舍意识。那我再追问一个，你这三个模块都是 LLM 驱动的，LLM 调用失败了你怎么办？整个检索还能返回结果吗？

### B：

这个我有完整的兜底链。每个模块都有独立的兜底，不会因为一个模块失败导致整个检索挂掉。

查询分解失败，我用正则提取硬过滤条件。正则能识别"X年以上"、"XK以内"、"会X的"这些模式，虽然不如 LLM 精确，但是基本条件能提取出来，sub_queries 就为空，走原有的 strategy_selector 改写逻辑。

多路召回的 Route C，MongoDB 查询失败的话，跳过这一路，不影响 Route A 和 Route B 的结果。只是少了结构化保底的召回。

上下文压缩失败，回退到取最高分 chunk 的 parent_content，就是原来的逻辑，LLM 评分还是能正常做，只是上下文信息少一些。

最后 LLM 评分失败，我还有两级兜底。第一级，LLM 调用失败，4 个维度全部回退到 rerank_score 乘以 100。第二级，LLM 返回了但是 JSON 解析失败，4 个维度全 0，reason 写"评分暂不可用"。

所以整个链路是层层兜底的，任何一环失败都有降级方案，保证最终能返回结果，不会白屏。

## 第四阶段：评分机制与 Agent 工作流（约 10 分钟）

### A：兜底做得挺完善。那我们聊聊评分。你刚才说 4 维度评分，能详细讲一下这个设计吗？为什么是 4 个维度，不是 3 个或者 5 个？

### B：

好的。4 维度是 skill、experience、education、salary 这四个。

选这四个维度主要是参考了招聘行业的一个通用评估框架。HR 评估一个候选人，最看重的就是这四个方面：技能匹配不匹配，经验够不够，学历达不达标，薪资期望合不合理。这四个维度基本覆盖了招聘决策的核心因素。

没有选 3 个是因为少了任何一个都不完整。少了 salary，HR 不知道这个人的薪资期望跟预算匹不匹配；少了 education，有些岗位有学历硬要求；少了 experience，初级岗位和高级岗位区分不开。

没有选 5 个或者更多，是因为更多维度会让 LLM 评分更难，而且维度之间会有重叠。比如如果加一个"项目经验"维度，它跟 experience 和 skill 都有重叠，LLM 评分的时候会混淆。4 个维度是相互独立又覆盖核心的最佳平衡。

每个维度 0 到 100 分，overall 是代码加权的，0.4 乘 skill 加 0.3 乘 experience 加 0.2 乘 education 加 0.1 乘 salary。这个权重也是参考招聘行业的，技能和经验更重要，学历和薪资权重低一些。overall 我特意用代码加权而不是让 LLM 直接给，是因为 LLM 给的 overall 有时候会跟 4 个维度的分不匹配，比如 4 个维度都是 80 分，overall 给个 60 分，明显不一致。代码加权保证一致性。

### A：你这里说 LLM 给的 overall 可能不一致，那你为什么不直接让 LLM 只输出 4 个维度，overall 完全用代码算？为什么还要 LLM 给 overall？

### B：

这个其实是历史原因。最早的设计是 LLM 只给 4 个维度，overall 完全代码算。后来我发现 LLM 有时候能给出更合理的 overall，比如某个候选人技能分高但是薪资期望远超预算，LLM 会把 overall 压低，但是代码加权算出来还是很高。所以我让 LLM 也给 overall，但是代码里做了兜底，如果 LLM 没给 overall 或者给的不合理，就用代码加权。

现在我的逻辑是，优先用 LLM 给的 overall，如果 LLM 没给，用代码加权 0.4/0.3/0.2/0.1 算。这样既利用了 LLM 的判断力，又有代码兜底保证一致性。

不过这个设计确实有点复杂，如果重做的话，我会让 LLM 只给 4 个维度，overall 完全代码算，这样更可控。LLM 的判断力可以通过 reason 字段体现，reason 是自然语言的推荐理由，LLM 可以在理由里说"虽然技能匹配但是薪资期望过高"，这样信息也不会丢。

### A：嗯，有反思。那 LLM 评分你用 JSON 模式输出，这个是怎么保证 LLM 一定返回合法 JSON 的？

### B：

用了几个手段。第一个是 response_format 参数，我传 response_format={"type": "json_object"}，这是 OpenAI 兼容接口支持的，强制 LLM 输出 JSON 格式。通义千问 qwen-plus 支持这个参数。

第二个是 system prompt 里明确要求返回 JSON，并且给出示例。prompt 里写"请返回 JSON 格式，不要有任何其他文字"，还给了具体的 JSON schema 示例。

第三个是 system prompt 里禁止返回 null，每个字段必须有 0 到 100 的数值。这个是为了避免 LLM 偷懒返回 null 导致后续代码报错。

第四个是代码层兜底。即使前面三道防线都失效，LLM 还是返回了不合法的 JSON，我用 try except 包住 json.loads，解析失败就走全 0 兜底，不会崩溃。

所以是 prompt 约束加 API 参数加代码兜底，三层保证。

### A：好。那我们聊聊 Agent 这块。你用 LangGraph 做 5 类意图识别，能讲一下为什么要做意图识别，而不是所有 query 都走检索？

### B：

好的。意图识别主要是为了优化体验和节省资源。

从体验角度，如果 HR 就是想闲聊一下，说"你好"或者"谢谢"，系统还去跑一遍检索然后返回一堆候选人，体验很怪。或者 HR 问"系统怎么用"，这是个问答，不需要检索候选人，直接回答就行。

从资源角度，检索链路涉及向量编码、Milvus 检索、Reranker 精排、LLM 评分，每一步都有成本，特别是 LLM 调用。如果闲聊也走检索，就是浪费 API 调用和计算资源。

所以我做了 5 类意图：chitchat 闲聊，search 搜索候选人，detail 查询已推荐候选人的详情，compare 对比候选人，qa 问答。每类意图走不同的处理路径。chitchat 和 qa 不检索，直接 LLM 回答；search 走完整检索链路；detail 和 compare 复用历史推荐的候选人，不重复检索。

意图识别用 LLM 一次调用，prompt 里告诉它 5 个意图的定义和示例，让它返回意图名。兜底是识别失败的时候默认走 qa，保证用户问题总能得到回答。

### A：你这里说 detail 和 compare 复用历史候选人，这个上下文是怎么管理的？多轮对话里怎么知道"前两个候选人"指的是谁？

### B：

这个是通过 AgentState 里的 last_candidates 字段管理的。每次 search 意图检索完，把推荐的候选人列表存到 state 的 last_candidates 里，同时存到 MongoDB 的会话历史里。

下一轮对话进来的时候，我先从 MongoDB 加载最近 5 轮历史，把上一轮的候选人列表提取出来作为 last_candidates。如果用户说"对比前两个"，意图识别成 compare，我不触发新的检索，直接用 last_candidates 的前两个做对比。

这里有个细节是姓名匹配。用户可能说"对比李志鹏和温佳蕊"，我需要从 last_candidates 里找到这两个名字对应的候选人。我用了两轮匹配算法，第一轮全名子串查找，第二轮歧义消除。比如用户说"李志"，但是 last_candidates 里有"李志"和"李志鹏"两个人，我会保留长名"李志鹏"，因为短名是长名的子串，用户大概率说的是更具体的那个。

这个 last_candidates 的设计让多轮对话很自然，HR 可以先搜，再问详情，再对比，再问为什么，全程不用重新检索，上下文连贯。

### A：SSE 流式响应你是怎么实现的？前端怎么接收？

### B：

后端用 FastAPI 的 StreamingResponse，media_type 设成 text/event-stream。我在 AgentService 里写了一个 async generator，每产生一个事件就 yield 一个符合 SSE 格式的字符串，格式是 "event: xxx\ndata: json字符串\n\n"。

事件类型有 8 种：intent 意图识别完，rewrite 查询改写完，retrieval 检索完，rank 精排完，candidates 候选人卡片，token LLM 流式输出的每个 token，done 全部完成，error 异常。

前端不能用 EventSource 接收，因为 EventSource 只支持 GET 请求，但是我发送消息是 POST。所以我用 fetch 加 ReadableStream 手动解析 SSE 格式。收到一个 chunk 就按 "event: xxx\ndata: xxx\n\n" 的格式切分，解析出事件类型和数据，然后分发到对应的处理函数。token 事件就追加到聊天界面，candidates 事件就渲染候选人卡片，done 事件就标记完成。

为什么用 SSE 不用 WebSocket，主要是因为 SSE 是单向推送，对 LLM 流式场景够用了，而且基于 HTTP，穿过代理和 CDN 更友好，不需要协议升级。WebSocket 是双向通信，对这个场景过重。

### A：你 SSE 流式的时候，如果 LLM 中途出错了，前端怎么处理？

### B：

后端这边，整个 stream_chat 方法用 try except 包住。如果 LLM 流式调用中途出错，我会 yield 一个 error 事件，data 里带错误码和错误信息。然后停止 yield，结束流。

前端这边，收到 error 事件就在聊天界面显示一个错误提示，比如"生成回复时出错，请重试"。同时把流式状态标记为结束，停止 loading 动画。

还有一个边界情况是网络断开。SSE 是长连接，如果网络断了，前端的 fetch 会抛异常，我在 catch 里处理，显示"连接断开"的提示，并且保留已经接收到的部分内容，不清空。

## 第五阶段：工程化与部署（约 5 分钟）

### A：好，技术细节聊得差不多了。我们聊聊工程化。你刚才说 464 个测试，能讲一下你的测试策略吗？特别是 RAG 这种链路，怎么写测试？

### B：

好的。我的测试分三层：单元测试、集成测试、E2E 测试。

单元测试主要测每个函数和类的独立逻辑。比如 _regex_extract_filters 这个函数，我写了各种 query 的测试用例，验证能不能正确提取学历、年限、薪资、技能条件。还有 chunker 的切块逻辑、pii 的脱敏逻辑、skill_synonyms 的扩展逻辑，这些纯函数都有单元测试。

集成测试主要测 service 层。比如 SearchService.search 方法，我会 mock 掉 LLM 调用和 Milvus 检索，验证整个检索流程的编排逻辑对不对，过滤条件有没有正确应用，去重有没有生效。这里 mock 有个技巧，motor 的 collection find 方法返回的是 cursor，不能用 AsyncMock，要用 MagicMock 加上 to_list 的 AsyncMock，不然会出 bug。

E2E 测试是最有价值的部分。我写了 32 步的招聘闭环测试，从登录开始，上传简历、列表查询、检索、对话、JD 匹配、面试、对比、导出邮件、看板，最后校验响应格式。数据在 9 个 API 路由间真实流转，上传的简历会被检索到、被推荐、被对比、被发邮件。

为了不依赖真实 MongoDB 和 Milvus，我自建了 FakeMongoDB 和 FakeRedis。FakeMongoDB 支持常用操作符，$set、$push、$in、$gte、$regex、$or 这些，让数据能真实流转。这个自建 mock 挺花时间的，但是让 E2E 测试很扎实。

关于 RAG 链路的测试，我的策略是分层 mock。LLM 调用全部 mock，返回预设的 JSON。Milvus 检索 mock，返回预设的 chunks。这样我能控制输入，验证每个环节的逻辑。比如查询分解，我 mock LLM 返回特定的 sub_queries 和 structured_filters，然后验证后面的多路召回有没有正确使用这些值。

### A：你提到 FakeMongoDB 自建 mock，这个工作量不小。为什么不直接用内存数据库，比如 mongomock 这个库？

### B：

其实我一开始考虑过 mongomock，但是有几个问题。一个是 mongomock 对一些操作符支持不完整，特别是 aggregate 的某些阶段，我的看板统计用了不少 aggregation。另一个是 mongomock 的行为跟真实 MongoDB 还是有差异，有些边界情况不一致，测试通过了但是真实环境挂。

所以我决定自建。自建的好处是我完全控制行为，需要什么操作符就实现什么，跟真实 MongoDB 行为对齐。坏处是工作量确实大，而且维护成本高，新增操作符要自己实现。

权衡下来，我觉得自建更适合我这个项目，因为我的查询模式比较固定，不需要支持所有 MongoDB 操作。如果是更通用的项目，用 mongomock 或者直接起一个内存 MongoDB 实例可能更合适。

### A：OK。那部署和可观测性你做了哪些？

### B：

部署用的是 Docker Compose，编排了 7 个容器：MongoDB、Redis、MinIO、Milvus、etcd、后端、前端。etcd 是 Milvus 的依赖，存元数据。

镜像构建用多阶段，前端是 Node 20 构建然后 Nginx 运行，后端是 Python 3.12-slim 精简镜像。docker-compose 里配了依赖顺序、健康检查、数据持久化 volume。MinIO 还有一个 init 容器，自动创建 bucket，不用手动操作。

可观测性做了四个层面。日志用 loguru，开发环境彩色人类可读格式，生产环境 JSON 结构化输出，方便 ELK 或者 Loki 采集。每个请求有一个 trace_id，贯穿日志和响应头，全链路追踪。

指标用 Prometheus，暴露 /metrics 端点，采集 QPS、延迟、在途请求数、简历解析数、LLM 调用数这些。还写了一个中间件记录每个请求的耗时，超过 200 毫秒的输出慢请求日志。

健康检查做了深度检查，/health/ready 端点会并发探测 MongoDB、Redis、MinIO、Milvus 的连通性，每个探测 3 秒超时，避免单个依赖卡住整个就绪检查。

错误追踪集成了 Sentry，未捕获的异常自动上报。前端也做了错误监控，JS 错误、Promise rejection、资源加载错误都上报到后端的 /api/v1/monitor/error 接口，有 500 毫秒节流防刷屏。

### A：你提到性能优化，BGE-M3 和 Reranker 这些大模型加载很慢，你怎么处理启动速度的？

### B：

这个是个关键优化。BGE-M3 大概 2GB，Reranker 大概 1GB，如果启动的时候同步加载，FastAPI 要几十秒才能就绪。

我把这些大模型的加载改成了异步预热。在 FastAPI 的 lifespan 启动事件里，用 asyncio.to_thread 把模型加载放到后台线程池，应用本身不等模型加载完就启动，秒级就绪。模型在后台继续加载，加载完之前如果有请求进来，第一次调用会等模型加载完，但是应用本身是活的。

还有 Redis 连接预热。Redis 连接池第一次请求的时候才建立 TCP 连接，有大概 2 秒延迟。我在启动的时候主动执行一个 PING 命令，强制建立连接，这样第一次真实请求就不用等了。优化前 /auth/me 首次请求 2039 毫秒，优化后 4 毫秒。

MongoDB 也建了索引，resumes 集合对 user_id、parse_status、created_at、education_level 加 work_years 建了复合索引，列表查询稳定在 5 到 15 毫秒。

## 第六阶段：反思与反问（约 2 分钟）

### A：好，技术问题问得差不多了。最后问一个开放性问题，你觉得这个项目还有什么不足？如果给你更多时间，你会优先改进什么？

### B：

嗯，这个我有想过几个方面。

第一个是 Milvus 字段命名的技术债。我 Collection 里有个字段叫 candidate_id，但是实际存的是 resume_id。这是因为早期简历模型没有 candidate_id 字段，直接用 resume_id 当唯一标识。后来引入 candidate_id 支持同一候选人多份简历，但是 Milvus 字段名没改，通过 chunk_map 做映射转换。这个应该重构，直接用 candidate_id，去掉映射层。

第二个是评分权重可配置化。现在 overall 加权权重 0.4/0.3/0.2/0.1 是硬编码的，应该抽到 .env 或者数据库，让 admin 可调。不同岗位、不同公司对四个维度的权重可能不一样。

第三个是同义词扩展自动化。现在是我手工维护映射表，覆盖面有限。应该用 LLM 动态扩展加缓存，这个我刚才也提到了。

第四个是 GPU 推理加速。现在 BGE-M3 和 Reranker 都在 CPU 上跑，延迟偏高。Docker 部署已经就绪，后续接 GPU 节点能大幅降低延迟。

如果优先级排序的话，我会先做评分权重可配置化，这个改动小但是价值大；然后做同义词扩展自动化，提升检索质量；Milvus 重构和 GPU 加速放到后面，因为改动大且当前能用。

### A：OK，反思挺到位的。我的问题问完了，你有什么想问我的吗？

### B：

嗯，我有两个问题想请教一下。

第一个是关于 RAG 的，您刚才有提到您做过 RAG 落地，我想问一下在您经验里，RAG 项目落地最大的挑战是什么？是检索质量、延迟、还是别的？

第二个是关于职业发展的，如果我想往 AI 工程师方向走，您觉得我现在这个项目展现的能力够吗？还有哪些技能是我应该补的？

### A：这两个问题问得不错。RAG 落地最大的挑战其实是评估，就是怎么量化检索质量。很多团队做 RAG 都是凭感觉调，没有系统的评估指标，比如召回率、精确率、NDCG 这些，导致优化没有方向。你能从 bug 驱动去优化，已经比很多人强了，但是缺一个系统的评估体系。至于职业发展，你这个项目已经展现了全栈和 AI 应用能力，缺的可能是模型训练和微调这块，如果有精力可以补一下。好，今天就到这里，感谢你的时间。

### B：

好的，谢谢面试官，收获很大。

## 面试复盘要点（给读者的自我检查清单）

### 回答得好的地方

1. **从 bug 驱动优化**：NLP 召回不足的真实案例，体现问题发现和解决能力
2. **trade-off 意识**：子查询数量限制、权重选择、缓存策略都有权衡说明
3. **兜底机制完整**：每个 LLM 调用都有降级方案，体现工程鲁棒性
4. **有反思**：主动提 Milvus 字段命名技术债、评分权重硬编码等不足
5. **能讲清原理**：Hybrid 检索、Reranker、父子块、Small-to-Big 都讲清了 why

### 可以改进的地方

1. **评估体系缺失**：没有系统的检索质量评估指标，优化凭经验
2. **同义词扩展手工维护**：有意识到局限，但解决方案还在 TODO
3. **A/B 测试不严谨**：权重选择说了"试过几个组合"，但不是系统实验
4. **缓存失效策略简单**：TTL 自然过期，没有主动失效，高频更新场景有问题

### 高频追问点（面试官可能继续深挖）

1. **RAG 评估**：有没有用 Ragas、TruLens 这类评估框架？怎么量化检索质量？
2. **向量数据库选型**：为什么 Milvus 不用 Qdrant、Weaviate、PGVector？
3. **Embedding 更新**：简历更新了，向量怎么增量更新？全量重建？
4. **多租户**：如果多个公司用这个系统，数据隔离怎么做？
5. **大模型成本**：LLM 调用频次高，成本怎么控制？有没有考虑本地模型？
6. **冷启动**：新上传的简历还没向量化，用户搜不到怎么办？
