## 一、项目概述

### 1.1 业务目标

为 HR 招聘场景提供基于 RAG 的智能简历推荐系统，核心价值：

1. **自然语言招聘**：HR 用自然语言描述需求（"5年Java后端，本科以上，30K以内"），系统智能推荐匹配候选人
2. **简历资产沉淀**：上传简历自动解析、去重、脱敏、入库，形成可检索的人才库
3. **招聘效率增强**：自动发邮件、Excel 导出、JD 匹配、面试题生成等实用工具
4. **数据洞察**：人才库结构化统计分析，辅助招聘决策

### 1.2 目标用户

| 角色 | 权限 | 主要场景 |
|------|------|---------|
| **HR** | 全部功能 | 对话推荐、简历管理、邮件发送、看板查看 |
| **HR 主管** | 全部功能 + 配置管理 | 同 HR + 邮件 SMTP 配置 |

> v2.5 采用 JWT 简单认证，不做 RBAC 三层权限（见 [MVP-Design.md](./MVP-Design.md) 弱化清单）。

### 1.3 核心业务流程

```
HR上传简历 → 系统解析/去重/脱敏/入库
                              ↓
HR对话/直接检索/JD匹配 ← 系统混合检索+改写+精排+评分
                              ↓
                    候选人卡片展示（评分+理由）
                              ↓
        ┌──────────┬──────────┬──────────┐
        ↓          ↓          ↓          ↓
     查看详情   打标签/收藏  发邮件    导出Excel
        ↓
   生成面试题 / 记录面试评价
```

## 三、功能验收标准详述

### F01 认证模块

#### 用户故事
作为 HR，我希望通过账号密码登录系统，以便安全地使用招聘功能。

#### 业务规则
- BR1.1：用户名密码校验通过后签发 JWT access_token（1小时）+ refresh_token（7天）
- BR1.2：除登录接口外，所有接口需携带有效 Token
- BR1.3：Token 失效后可用 refresh_token 刷新，无需重新登录
- BR1.4：登出后当前 Token 加入 Redis 黑名单，立即失效

#### 验收标准
- **[P0] AC1.1**：正确账号密码登录，返回 access_token + refresh_token + user 信息
- **[P0] AC1.2**：错误密码登录，返回 code=1001，提示"用户名或密码错误"
- **[P0] AC1.3**：未携带 Token 访问受保护接口，返回 code=1002
- **[P0] AC1.4**：携带过期 Token，返回 code=1002，前端跳转登录页
- **[P1] AC1.5**：使用 refresh_token 成功换取新 access_token
- **[P0] AC1.6**：登出后，原 Token 访问任意接口返回 code=1002

#### 边界与异常
- 密码错误连续 5 次：v2.5 不做锁定（简化），仅提示错误
- refresh_token 过期：需重新登录

### F02 简历上传与解析

#### 用户故事
作为 HR，我希望上传 PDF/DOCX/图片简历，系统自动解析结构化信息并入人才库，以便后续检索推荐。

#### 业务规则
- BR2.1：支持格式 `.pdf` `.docx` `.png` `.jpg`，单文件 ≤ 10MB
- BR2.2：解析流程为两级降级：PyMuPDF/python-docx 提取文本 → 失败则 RapidOCR（不做三级 Unstructured）
- BR2.3：LLM 结构化提取字段：姓名、手机、邮箱、性别、年龄、学历、工作年限、技能、工作经历、教育经历、期望薪资
- BR2.4：手机号/邮箱脱敏存储（138****1234 / zhang***@xx.com），同时存 phone_hash/email_hash 用于去重
- BR2.5：去重逻辑——上传时计算 phone_hash + email_hash，命中已有记录则标记 `is_duplicate=true`，返回 `duplicate_with` 指向已有简历
- BR2.6：父子块切分——子块 300 字符入 Milvus，父块 1200 字符作为 parent_content 一起存（Small-to-Big 还原）
- BR2.7：解析异步进行，上传后立即返回 `parse_status=parsing`，前端轮询状态
- BR2.8：原始文件存 MinIO，结构化数据存 MongoDB，向量存 Milvus

#### 验收标准
- **[P0] AC2.1**：上传 PDF 简历，返回 resume_id + parse_status=parsing
- **[P0] AC2.2**：解析完成后，parse_status=completed，MongoDB 有完整结构化数据，Milvus 有向量
- **[P0] AC2.3**：上传重复简历（相同手机号），返回 is_duplicate=true + duplicate_with
- **[P0] AC2.4**：上传图片型简历，走 OCR 路径成功解析
- **[P0] AC2.5**：上传损坏文件，parse_status=failed，返回 code=2001 错误信息
- **[P0] AC2.6**：手机号在详情页显示为 138****1234（脱敏），不显示明文
- **[P1] AC2.7**：上传 DOCX 简历解析成功
- **[P1] AC2.8**：overwrite=true 时，重复简历覆盖更新而非标记重复
- **[P2] AC2.9**：薪资字符串"20-30K"正确解析为 {min:20, max:30}

#### 边界与异常
- 文件 > 10MB：拒绝上传，提示"文件过大"
- 不支持的格式：拒绝上传，提示"仅支持 PDF/DOCX/PNG/JPG"
- LLM 提取失败：标记 parse_status=failed，不影响其他简历
- MinIO 存储失败：回滚，返回 500 错误

#### 性能要求
- 单份 PDF 解析（含 LLM 提取 + 向量化）：≤ 30 秒
- 并发上传：支持 5 个并发解析任务

### F03 简历列表与筛选

#### 用户故事
作为 HR，我希望按关键词、标签、学历、年限、薪资等条件筛选简历列表，以便快速找到目标候选人。

#### 业务规则
- BR3.1：分页查询，默认 page=1, page_size=20，最大 100
- BR3.2：支持筛选条件：关键词（姓名/技能模糊）、标签精确、收藏、最低学历、最低年限、薪资范围、解析状态
- BR3.3：多条件之间为 AND 关系
- BR3.4：列表项字段见 [API-Design.md 10.1 ResumeListItem](./API-Design.md)

#### 验收标准
- **[P0] AC3.1**：无条件查询返回分页列表，含 total/page/page_size/total_pages
- **[P0] AC3.2**：keyword="Java" 返回技能或姓名含 Java 的简历
- **[P0] AC3.3**：tag="已面试" 返回带该标签的简历
- **[P0] AC3.4**：is_favorite=true 仅返回收藏的简历
- **[P0] AC3.5**：education_min=2 返回硕士及以上（education_level≥2）
- **[P0] AC3.6**：salary_max=30 返回期望薪资最低值 ≤30K 的简历
- **[P1] AC3.7**：多条件组合筛选结果正确（AND 关系）
- **[P1] AC3.8**：分页越界返回空 list，不报错

#### 性能要求
- 列表查询响应：≤ 500ms（10万条数据量下，依赖 MongoDB 索引）

### F11 流式对话推荐（核心功能）

#### 用户故事
作为 HR，我希望用自然语言描述招聘需求，系统流式输出推荐结果和候选人卡片，体验流畅。

#### 业务规则
- BR11.1：意图类型 4 种：chitchat（闲聊）/search（搜索）/detail（详情询问）/compare（对比）
- BR11.2：search 意图触发完整检索链路：Query改写→混合检索→Reranker→LLM评分
- BR11.3：Query改写 4 种策略（复用 EduRAG）：
  - direct：明确 query 直接检索
  - hyde：模糊描述生成假设答案再检索
  - subquery：复杂多条件拆解子查询
  - backtracking：指代上轮内容简化回溯
- BR11.4：SSE 流式输出 8 种事件：intent/rewrite/retrieval/rank/token/candidates/done/error
- BR11.5：召回不足时（如 0 条），clarify 节点生成追问问题，不做 HITL 超时降级
- BR11.6：detail/compare 意图基于 last_candidates（上轮推荐）回答
- BR11.7：支持中断流式输出（前端 stop）

#### 验收标准
- **[P0] AC11.1**：发送"找个5年Java后端"，流式输出候选人推荐 + 理由
- **[P0] AC11.2**：SSE 事件按序触发：intent → rewrite → retrieval → rank → token(多个) → candidates → done
- **[P0] AC11.3**：候选人卡片含 score（0-100）+ reason（自然语言理由）
- **[P0] AC11.4**：发送"你好"，识别为 chitchat，直接回复不检索
- **[P0] AC11.5**：发送"第一个候选人详细说说"，识别为 detail，基于 last_candidates 回答
- **[P0] AC11.6**：发送"对比前两个候选人"，识别为 compare，输出对比
- **[P0] AC11.7**：发送"找个会写COBOL的"（库里无匹配），触发 clarify 追问
- **[P0] AC11.8**：流式过程中点击"停止"，立即中断输出
- **[P0] AC11.9**：LLM 调用失败，SSE 推送 error 事件，前端提示错误
- **[P1] AC11.10**：多轮对话上下文连贯（第3轮引用第1轮内容）
- **[P1] AC11.11**：模糊描述"找个Java大佬"触发 HyDE 策略

#### 边界与异常
- 空消息：拒绝发送，提示"请输入内容"
- 网络断开：SSE 中断，前端提示"连接断开"
- LLM 超时：tenacity 重试 3 次，仍失败推送 error 事件

#### 性能要求
- 首字输出延迟（TTFT）：≤ 3 秒
- 完整推荐响应：≤ 15 秒（含检索+评分+流式输出）
- 检索单次（不含 LLM 评分）：≤ 2 秒

### F13 智能检索

#### 用户故事
作为 HR，我希望直接输入检索条件快速搜索候选人，不通过对话，以便批量查找。

#### 业务规则
- BR13.1：与对话的 search 意图共享检索链路，但非流式直接返回
- BR13.2：strategy 可指定（direct/hyde/subquery/backtracking）或 auto（LLM 选）
- BR13.3：filters 硬条件过滤：学历/年限/薪资/技能
- BR13.4：结果缓存 5 分钟（Redis）

#### 验收标准
- **[P0] AC13.1**：query="Java后端" 返回候选人列表，含 score + reason
- **[P0] AC13.2**：filters education_min=2 仅返回硕士及以上
- **[P0] AC13.3**：filters salary_max=30 仅返回期望薪资 ≤30K
- **[P0] AC13.4**：相同 query+filters 第二次查询命中缓存，响应更快
- **[P1] AC13.5**：strategy=hyde 对模糊 query 召回更好
- **[P0] AC13.6**：top_k 限制生效，默认 10，最大 50

#### 性能要求
- 首次检索（含 LLM 评分）：≤ 5 秒
- 缓存命中：≤ 200ms

## 四、非功能性需求

### 4.1 性能要求

| 指标 | 标准 | 说明 |
|------|------|------|
| API 响应（普通） | ≤ 500ms | 列表/详情/标签等 |
| 检索响应（首次） | ≤ 5秒 | 含 LLM 评分 |
| 检索响应（缓存命中） | ≤ 200ms | Redis 缓存 |
| 对话首字延迟（TTFT） | ≤ 3秒 | SSE 首 token |
| 对话完整响应 | ≤ 15秒 | 含检索+评分+流式 |
| 简历解析 | ≤ 30秒/份 | 含 LLM 提取+向量化 |
| 看板统计 | ≤ 2秒 | aggregation+缓存 |
| 并发上传 | 5个并发 | 异步解析 |

### 4.2 可靠性要求

- **[P0]** LLM 调用失败时 tenacity 重试 3 次，仍失败返回明确错误码（2002）
- **[P0]** 简历解析失败不影响其他简历，标记 failed 状态
- **[P0]** 数据一致性：删除简历时 MinIO/MongoDB/Milvus 三处同步清理
- **[P1]** Redis 缓存失效不影响功能正确性（回源查询）
- **[P1]** MongoDB 单库备份（mongodump），不做三重备份

### 4.3 安全要求

- **[P0]** PII 脱敏：手机号/邮箱展示与导出均脱敏（138****1234）
- **[P0]** JWT 认证：除登录外所有接口需 Token
- **[P0]** 密码哈希存储（passlib bcrypt），不明文
- **[P0]** SMTP 密码不明文返回 API
- **[P1]** Token 黑名单：登出立即失效
- 注：v2.5 不做 AES-256 加密、RBAC、Guardrails 公平性检查（见 [MVP-Design.md](./MVP-Design.md) 弱化清单）

### 4.4 可观测性要求

- **[P0]** loguru 日志全链路，绑定 trace_id
- **[P0]** API 响应含 trace_id 字段，便于问题排查
- **[P1]** 关键节点日志：检索/评分/邮件发送等
- 注：v2.5 不做 OpenTelemetry/Prometheus/LangSmith（见 [MVP-Design.md](./MVP-Design.md) 弱化清单）

### 4.5 兼容性要求

- **前端**：Chrome 90+ / Edge 90+ / Firefox 90+
- **后端**：Python 3.11+
- **数据库**：MongoDB 7.0 / Milvus 2.4+ / Redis 7.0+

## 五、弱化功能说明（明确不做）

依据 [MVP-Design.md](./MVP-Design.md) 弱化清单，以下功能 v2.5 **明确不做**，验收时不作要求：

| 功能 | 不做原因 | 后续计划 |
|------|---------|---------|
| bbox 高亮遮罩 | 前端 Canvas 绘制复杂，收益有限 | 保留 PDF 预览即可 |
| LayoutLMv3 版面预处理 | 模型部署复杂，LLM 提取够用 | — |
| Guardrails 溯源校验 + 公平性检查 | 依赖 chunk_id 绑定，非核心 | — |
| HITL 60秒超时降级 | 降级链+Checkpointer 太复杂 | 用 clarify 直接追问 |
| OpenTelemetry 全链路 Tracing | 部署重，loguru 够用 | — |
| Prometheus + Grafana 监控 | 非核心 | 后续按需 |
| LLM 主备降级链 | 配置多 API 复杂 | 单 LLM + tenacity 重试 |
| PII AES-256 加密 | 密钥管理复杂 | 字符串脱敏够用 |
| 三层数据备份 | 备份脚本复杂 | MongoDB mongodump |
| RBAC 三层权限 | 过度设计 | JWT 简单认证 |

> 验收时，不得因上述功能缺失而判定不合格。

## 六、端到端验收场景

### 场景1：完整招聘推荐流程（核心场景）

1. HR 上传 5 份 Java 后端简历 → 全部解析成功
2. HR 上传第 6 份重复简历（相同手机号）→ 提示重复
3. HR 在对话页输入"找个5年Java后端，本科以上，30K以内"
4. 系统流式输出：意图识别 → Query改写 → 检索 → 评分 → 候选人卡片 + 推荐理由
5. HR 点击第一个候选人卡片 → 右侧抽屉展示详情 + PDF预览
6. HR 给候选人打标签"重点关注"并收藏
7. HR 点击"邮件"→ 发送推荐邮件给主管，含 Excel 附件
8. 主管收到邮件，打开 Excel 查看候选人列表

**验收**：全流程无报错，候选人推荐相关度高，邮件成功送达。

### 场景2：JD匹配流程

1. HR 在 JD 匹配页粘贴一段 Java 高级开发岗位 JD
2. 系统解析出要求：Java/Spring Boot/5年/本科
3. 返回匹配候选人 Top-10，每个候选人含 matched_skills + missing_skills 匹配分析
4. HR 选中第1名候选人，生成针对性面试题（technical/project 维度）
5. HR 面试后记录评价（rating=4, result=pass）
6. 在简历详情页可看到该面试评价记录

**验收**：JD 解析准确，匹配候选人排序合理，面试题针对技能栈，评价记录可追溯。

### 场景3：多轮对话上下文

1. HR 创建会话"前端招聘"
2. 第1轮："找个3年前端" → 推荐候选人 A/B/C
3. 第2轮："第一个会Vue3吗" → 基于 last_candidates(A) 回答详情
4. 第3轮："对比前两个" → 对比 A 和 B
5. 第4轮："还有类似的吗" → 基于上轮推荐相似候选人
6. 第5轮："你好" → chitchat 闲聊回复

**验收**：意图识别准确，detail/compare 正确引用上轮候选人，多轮上下文连贯。

### 场景4：数据看板

1. HR 主管进入数据看板页
2. 查看简历总数、本周新增、本月新增
3. 查看技能分布柱状图（Top-20）
4. 查看学历分布、工作年限分布饼图
5. 切换 range=week 仅看本周数据

**验收**：统计数据准确，图表正确渲染，时间筛选生效。
